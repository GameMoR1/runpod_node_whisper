# Документация по тестовому набору

Этот документ предоставляет обзор тестового набора для проекта Whisper CUDA FastAPI Service. Он объясняет стратегию тестирования, как запускать тесты, что покрывает каждый тестовый файл, ключевые шаблоны тестирования и фикстуры, а также рекомендации по расширению тестового набора.

## Обзор

Тестовый набор организован в две основные категории:

1. **Юнит-тесты**: Расположены в `tests/unit/`, эти тесты фокусируются на отдельных компонентах в изоляции, в основном на функционале реестра моделей.
2. **End-to-End (E2E) тесты**: Расположены в `tests/e2e/`, эти тесты проверяют полный рабочий процесс сервиса, от API-точек входа до фоновой обработки и обратных вызовов.

Тесты используют `pytest` с `pytest-asyncio` для асинхронного тестирования. Мокирование широко используется для изоляции системы под тестом от внешних зависимостей, таких как база данных, GPU и внешние HTTP-сервисы.

## Запуск тестов

Для запуска тестового набора убедитесь, что у вас установлены зависимости разработки. Затем выполните:

```bash
pytest
```

Вы можете запускать конкретные категории тестов:

```bash
# Запуск только юнит-тестов
pytest tests/unit/

# Запуск только E2E-тестов
pytest tests/e2e/

# Запуск конкретного тестового файла
pytest tests/e2e/test_callback.py
```

Конфигурация тестов определена в `pytest.ini`:
- `asyncio_mode = auto`: Включает автоматическую обработку асинхронных тестов.
- `asyncio_default_fixture_loop_scope = function`: Каждая тестовая функция получает новый цикл событий.
- `python_files = test_*.py`: Обнаруживает файлы, соответствующие этому шаблону.
- `testpaths = tests`: Ищет тесты в директории `tests`.

## Юнит-тесты

### `tests/unit/test_model_registry.py`

Этот файл тестирует класс `ModelRegistry`, который отвечает за загрузку информации о моделях из базы данных и управление загрузкой моделей.

**Ключевые тестовые случаи:**
- `test_load_enabled_models_from_db`: Проверяет, что реестр загружает только включенные модели из базы данных и готовит их к загрузке.
- `test_filter_only_enabled_models`: Обеспечивает, что загружаются только модели, помеченные как включенные в базе данных.
- `test_successful_model_download`: Тестирует, что модель может быть успешно загружена и ее статус обновлен.
- `test_model_download_error_with_retry`: Проверяет, что сбои загрузки повторяются настроенное количество раз перед пометкой модели как ошибочной.
- `test_parallel_download_two_models`: Проверяет, что несколько моделей могут быть загружены одновременно.
- `test_skip_already_downloaded_model`: Подтверждает, что модели, уже помеченные как известные, не загружаются повторно.

**Используемые фикстуры:**
- `mock_db_models`: Мокирует вызовы базы данных для возврата предопределенных строк моделей и включенных моделей.
- `parameters_model_tiny`, `parameters_model_base`: Предоставляют образцы объектов `ModelState` для тестирования.

## End-to-End (E2E) тесты

E2E-тесты имитируют реальное использование сервиса, выполняя HTTP-запросы к приложению FastAPI и проверяя ответы и фоновую обработку.

### `tests/e2e/test_endpoint.py`

Тесты базовой доступности и функциональности API-точек входа:
- `test_endpoints_availability`: Проверяет, что точки входа `/health`, `/queue`, `/status` и `/dashboard/state` доступны.
- `test_websocket_dashboard`: Проверяет, что WebSocket-точка входа для дашборда возвращает ожидаемые данные.

### `tests/e2e/test_callback.py`

Тесты полного рабочего процесса транскрипции от загрузки файла до обратного вызова:
- `test_full_scenario_upload_to_callback`:
  - Загружает аудиофайл для транскрипции.
  - Проверяет, что задача поставлена в очередь и обработана.
  - Проверяет, что задача завершилась успешно с правильными данными результата.
  - Убеждается, что обратный вызов доставлен успешно.
  - Подтверждает, что сервис возвращается в состояние простоя после обработки.

### `tests/e2e/test_error.py`

Тесты обработки ошибок на различных этапах процесса транскрипции:
- `test_upload_file_error`: Симулирует сбой загрузки файла и проверяет, что исключение распространяется.
- `test_transcription_error`: Мокирует сбой транскрипции и проверяет, что статус задачи помечен как неудачный с соответствующей ошибкой.
- `test_callback_error`: Симулирует сбой доставки обратного вызова и проверяет, что статус обратного вызова отражает ошибку.
- `test_callback_http_error`: Тестирует случай, когда конечная точка обратного вызова возвращает HTTP-ошибку статуса.

### `tests/e2e/test_language.py`

Тесты обработки языка и выбора модели:
- `test_russian_on_tiny_model`: Проверяет, что транскрипция на русском языке работает с моделью tiny.
- `test_english_on_base_model`: Проверяет, что транскрипция на английском языке работает с моделью base.
- `test_multilingual_detection`: Тестирует транскрипцию на нескольких языках (русский, английский, немецкий, французский) с использованием модели small.
- `test_auto_language_detection`: Проверяет, что при неуказании языка сервис по умолчанию использует русский.
- `test_model_size_impact_on_performance`: Проверяет, что большие модели требуют больше времени обработки, чем меньшие (tiny < base < small).

### `tests/e2e/test_parallel.py`

Тесты параллельной обработки задач:
- `test_parallel_jobs_submission`: Отправляет несколько задач транскрипции одновременно и проверяет, что они принимаются и ставятся в очередь/выполняются соответствующим образом.

## Шаблоны тестирования и фикстуры

### Ключевые шаблоны

1. **Мокирование внешних зависимостей**:
    - Библиотека `unittest.mock` используется для патчинга внешних сервисов, таких как база данных (`app.model_registry.fetch_all`), загрузка моделей Whisper (`whisper.load_model`), операции с GPU и HTTP-клиенты.
    - Это обеспечивает быстрые, детерминированные тесты, не требующие реальных GPU или внешних сервисов.

2. **Асинхронное тестирование**:
    - Тесты, использующие `pytest.mark.asyncio` и `pytest_asyncio.fixture` для асинхронного кода.
    - Фикстура `client` предоставляет `AsyncClient` для выполнения HTTP-запросов к приложению FastAPI.

3. **Проверка состояния**:
    - E2E-тесты часто опрашивают точку входа `/status` для ожидания завершения задачи, имитируя реальное поведение.
    - Они проверяют не только конечное состояние, но и промежуточные состояния (в очереди, в обработке).

4. **Проверка обратного вызова**:
    - Тесты проверяют, что обратные вызовы выполняются с правильными полезными нагрузками, включая информацию о времени и метриках GPU.

### Важные фикстуры (в `conftest.py`)

- `mock_db_models`: Предоставляет мок-ответы базы данных для данных моделей и включенных моделей.
- `parameters_model_tiny`, `parameters_model_base`: Образцы объектов `ModelState` для tiny и base моделей.
- `models_rows`, `enabled_rows`: Альтернативные фикстуры для строк базы данных.
- `app`: Предоставляет экземпляр приложения FastAPI.
- `client`: `AsyncClient`, настроенный для тестирования приложения.
- `test_audio_file`: Создает временный аудиофайл для тестирования загрузок.
- `mock_app_state`: Полностью замокированный экземпляр `AppState`, имитирующий готовый сервис с базовой моделью.
- `mock_app_state_with_multiple_models`: Аналогичен `mock_app_state`, но с tiny, base и small моделями.
- `data_model_*`: Фикстуры, предоставляющие примеры данных запроса для точки входа `/transcribe`.
- `files`: Предоставляет тестовый аудиофайл как multipart-загрузку.
- `test_transcription_text`: Пример результата транскрипции, используемый в моках.
- `texts_by_language`: Примеры текстов на нескольких языках для тестов обнаружения языка.
- `models_with_delays`: Сопоставление имен моделей с имитированными задержками обработки для тестов производительности.

## Расширение тестового набора

При добавлении новых функций или изменении существующих, учитывайте следующие рекомендации:

1. **Где добавлять тесты**:
    - **Юнит-тесты**: Для изолированной логики (например, утилитных функций, методов классов) -> `tests/unit/`
    - **E2E-тесты**: Для функций, связанных с API-точками входа, фоновой обработкой или обратными вызовами -> `tests/e2e/`

2. **Соглашения об именовании**:
    - Тестовые файлы: `test_<feature>.py`
    - Тестовые функции: `test_<scenario>_<expected_behavior>()`

3. **Использование фикстур**:
    - Повторно используйте существующие фикстуры из `conftest.py` по возможности.
    - Добавляйте новые фикстуры в `conftest.py`, если они будут использоваться в нескольких тестовых файлах.
    - Для фикстур, специфичных для файла, определяйте их непосредственно в тестовом файле.

4. **Лучшие практики мокирования**:
    - Мокируйте только внешние зависимости, с которыми взаимодействует тестируемый модуль.
    - Используйте `AsyncMock` для асинхронных методов.
    - Проверяйте вызовы моков при необходимости, чтобы убедиться в правильном взаимодействии.

5. **Структура E2E-тестов**:
    - Arrange: Настройте любые необходимые моки и тестовые данные.
    - Act: Выполните API-запрос(ы).
    - Assert: Проверьте немедленные ответы и затем опросите на асинхронное завершение.
    - Cleanup: Любая необходимая очистка обычно обрабатывается фикстурами.

6. **Производительные соображения**:
    - Делайте тесты быстрыми, минимизируя ненужное мокирование и используя имитации в памяти.
    - Избегайте реальных операций с GPU или загрузки моделей в тестах.

7. **Запуск тестов во время разработки**:
    - Используйте `pytest -x` для остановки при первой неудаче.
    - Используйте `pytest -vv` для подробного вывода.
    - Используйте `pytest -k <pattern>` для запуска тестов, соответствующих ключевому слову.

Следуя этим шаблонам и последовательно расширяя существующий тестовый набор, мы можем поддерживать высокое покрытие тестами и надежность по мере развития проекта.
