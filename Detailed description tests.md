## Документация и целевое описание к тестам проекта runpod_node_whisper

### 1. test_model_registry - класс модульных тестов

+ **async def test_load_enabled_models_from_db(mock_db_models)** - Тест проверяет корректность инициализации реестра моделей при загрузке данных из базы. Берем фикстуру Берем фикстуру mock_db_models и имитируем наличие в базе данных нескольких записей о моделях (tiny, small, base, medium), где только часть из них имеет флаг включения. Проверяем, что объект класса ModelRegistry после выполнения метода загрузки содержит только активные модели с корректно заполненными атрибутами (идентификатор, имя, статус). Далее ожидаем, что количество вызовов функций получения данных и подготовки файлов строго соответствует количеству активных моделей в предоставленной выборке.


+ **async def test_filter_only_enabled_models(mock_db_models)** - Проверка строгого соблюдения фильтрации моделей по их статусу в базе данных. В ходе теста иммитируется ситуация, когда в общем справочнике присутствуют четыре модели, но в таблице настроек активными числятся только две. Программа выполняет итерацию по всем загруженным в ModelRegistry объектам и проверяет, что атрибут «enabled» каждого объекта имеет значение True. Далее подтверждается, что итоговое количество загруженных моделей в реестре в точности соответствует количеству активных записей в тестовой выборке, что гарантирует отсутствие нецелевых или отключенных моделей в оперативной памяти сервера.


+ **async def test_successful_model_download(parameters_model_tiny)** - Берем фикстуру parameters_model_tiny и проверяем жизненный цикл процесса загрузки модели Whisper и корректность обновления её внутреннего состояния. В начале теста объект модели инициализируется со статусом «в очереди на загрузку» (queued_for_download) и нулевым прогрессом. С помощью механизма подмены системных модулей (sys.modules patching) иммитируется вызов внешней библиотеки Whisper. Тест подтверждает, что метод загрузки вызывается с корректными параметрами конфигурации: целевое устройство (CPU), идентификатор модели и путь к локальному кэшу. В финальной фазе проверяется переход модели в статус «загружено» (downloaded) с установкой стопроцентного прогресса и отсутствием ошибок выполнения.


+ **async def test_model_download_error_with_retry(parameters_model_tiny)** - Проверяется механизм обработки исключений и логика повторных попыток (retry policy) при возникновении критических ошибок во время загрузки модели. Путем внедрения фиктивного исключения «Connection error» в вызываемый метод библиотеки Whisper, имитируется нестабильное сетевое соединение. Тест подтверждает, что система выполняет ровно три попытки возобновления загрузки перед окончательным признанием операции неудачной. В завершении проверяется корректность записи диагностической информации: объекту модели присваивается статус «error», обнуляется прогресс выполнения и сохраняется текст ошибки для последующего анализа пользователем.


+ **async def test_parallel_download_two_models(parameters_model_tiny, parameters_model_base)** - Тест проверяет возможность параллельной загрузки нескольких моделей машинного обучения без взаимной блокировки процессов. Берутся параметры моделей `tiny` и `base`, дальше с помощью метода asyncio.gather имитируется одновременный запуск задач по скачиванию весов для моделей «tiny» и «base». Для оптимизации скорости выполнения теста применяется мокирование функции asyncio.sleep и ограничение количества попыток повторного соединения. Тест подтверждает, что внешняя библиотека загрузки вызывается независимо для каждой модели с соответствующими параметрами. В результате проверяется, что обе модели успешно и независимо друг от друга перешли в конечное состояние «downloaded», зафиксировав полный прогресс выполнения без возникновения ошибок или конфликтов доступа к общим данным реестра.


+ **async def test_skip_already_downloaded_model(models_rows, enabled_rows)** - Проверка логики предотвращения избыточных операций загрузки для уже существующих в системе моделей. С помощью мокирования метода «is_model_known» имитируется ситуация, при которой все активные модели из базы данных уже были успешно загружены ранее и присутствуют в локальном кэше. Проверяется выполнение процедуры инициализации реестра «load_from_db_and_prepare». Тест подтверждает, что при обнаружении существующих файлов моделей система корректно пропускает этап обращения к методу «_download_model», что гарантирует отсутствие повторных сетевых запросов и экономию дисковых ресурсов при перезапуске приложения.


### 2. test_callback.py - полный сценарий e2e с callback

+ **async def test_full_scenario_upload_to_callback(
        client: AsyncClient,
        test_audio_file: Path,
        mock_app_state: AppState,
        data_model_base)** - тестирование сквозного бизнес-процесса обработки медиафайла в асинхронном режиме. Процедура включает в себя: отправку файла через POST-запрос, валидацию постановки задачи в очередь (статус busy), циклическое ожидание обработки через эндпоинт статуса и финальную проверку структуры результата. Особое внимание уделяется корректности возвращаемых метрик: проверяется наличие текста транскрибации, временных сегментов и показателей производительности GPU (VRAM, утилизация). Завершающим этапом контролируется успешность имитации срабатывания callback-механизма и возврат системы в исходное состояние готовности (idle/ready). 


### 3. test_error.py - тестирование обработки ошибок

+ **async def test_upload_file_error(
        client: AsyncClient,
        test_audio_file: Path,
        mock_app_state: AppState,
        data_model_base,
        files)** - Негативный тест, который проверяет поведение системы при возникновении критической ошибки файловой системы во время загрузки медиаданных. С помощью подмены (patching) асинхронного метода anyio.open_file имитируется сбой операции записи файла на диск (например, из-за отсутствия прав доступа или переполнения хранилища). Проверяется, что при возникновении исключения «File upload failed» процесс транскрибации корректно прерывается на этапе приема файла. Тест подтверждает способность системы идентифицировать ошибки ввода-вывода, предотвращая дальнейшую обработку некорректно загруженных данных.


+ **async def test_callback_error(
        client: AsyncClient,
        test_audio_file: Path,
        mock_app_state: AppState,
        data_model_base,
        files,
        test_transcription_text
)** - Проверяет корректность обработки критических сбоев непосредственно в процессе нейросетевой расшифровки (Whisper). С помощью внедрения исключения «Transcription failed» в метод transcribe имитируется программный сбой модели при обработке аудио. Проверяется жизненный цикл асинхронной задачи: от успешного приема API (статус 200) до перехода в терминальное состояние «failed» в очереди. Тест подтверждает, что система перехватывает исключения внутри воркера, корректно транслирует текст ошибки в статус задачи и гарантирует отсутствие ложных результатов в ответе. Это подтверждает надежность механизмов отслеживания состояния задач при возникновении непредвиденных ошибок в вычислительном коде.


+ **async def test_callback_error(
        client: AsyncClient,
        test_audio_file: Path,
        mock_app_state: AppState,
        data_model_base,
        files,
        test_transcription_text
)** - Тест на проверку отказоустойчивости системы при сбое внешней подсистемы уведомлений (Callback). Сценарий имитирует успешное выполнение транскрибации аудиофайла нейросетью Whisper, но инициирует искусственное исключение «Callback failed» в момент попытки отправить результат на указанный пользователем callback_url. Тест подтверждает, что статус основной задачи корректно переходит в состояние «completed» (завершено), работа не аннулируется, однако в метаданных задачи флаг доставки «delivered» устанавливается в значение False, а в поле ошибки записывается причина неудачной отправки. Это гарантирует сохранение вычислительного результата в системе даже при временной недоступности принимающей стороны.


+ **async def test_callback_http_error(
        client: AsyncClient,
        test_audio_file: Path,
        mock_app_state: AppState,
        data_model_base,
        files,
        test_transcription_text
)** - Этот тест проверяет сценарий, при котором внешний сервис (принимающий callback) отвечает ошибкой (HTTP 500), а не падает по таймауту или сетевому сбою. Это тонкая настройка: мы проверяем, как наше приложение реагирует на корректный, но «негативный» ответ от сервера клиента. В сценарии имитируется ситуация, когда удаленный сервер доступен, но возвращает ошибку «500 Internal Server Error». Тест подтверждает, что основная задача транскрибации сохраняет статус «completed», так как данные успешно обработаны, однако подсистема уведомлений корректно классифицирует ответ сервера как неудачный. Проверяется установка флага «delivered» в значение False и наличие записи об ошибке в метаданных задачи. Это гарантирует, что пользователь получит информацию о том, что автоматическое уведомление не удалось из-за проблем на стороне принимающего сервера.

 + **async def test_endpoints_availability(client_is_endpoint, data_model_tiny, files)** - Данный набор тестов выполняет проверку доступности (Smoke testing) всех ключевых публичных интерфейсов сервера. Тест гарантирует, что базовые эндпоинты (/health, /queue, /dashboard/state) возвращают корректные коды ответов, а механизмы обработки ошибок (например, 404 для несуществующих задач) работают согласно спецификации.


+ **def test_websocket_dashboard()** - Тест подтверждает, что сервер автоматически транслирует начальное состояние системы (Initial State) сразу после установления связи. Проверяется наличие в JSON-пакете ключевых узлов данных: списка состояний моделей (models) и общих метрик здоровья сервера (health). Это гарантирует, что фронтенд-интерфейс мониторинга получит все необходимые данные для корректной визуализации текущей нагрузки без выполнения дополнительных HTTP-запросов.

С помощью функции asyncio.gather имитируется одновременное обращение нескольких клиентов к API с требованиями использовать разные вычислительные модели. Тест подтверждает способность сервера асинхронно генерировать идентификаторы задач и корректно управлять состоянием очереди в условиях конкурентного доступа. Финальная часть сценария проверяет целостность данных в эндпоинте мониторинга очереди, гарантируя, что информация о задачах в статусах «ожидание» и «исполнение» доступна и структурирована верно даже при высокой плотности запросов. Это подтверждает готовность API к работе в многопользовательской среде и корректность интеграции асинхронного веб-фреймворка с фоновым планировщиком задач.

Подтверждается, что система правильно маршрутизирует задачу на выбранную модель, корректно обрабатывает UTF-8 символы и возвращает сегментированный результат, соответствующий исходному аудио.
